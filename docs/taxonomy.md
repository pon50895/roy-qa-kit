# 測試分類（Coverage Model）

每個功能/票，過一遍這 6 類，判斷各要不要案例。Playwright 用 tag 標：`@type @value @edge @logic @file @sanity @integration`。

---

## 1. 類型測試 `@type`
欄位/參數只接受正確**型別**，錯型別要擋。
- 前端Web：出生年=數字、手機=數字、Email=email、身分證=英數格式、代碼=12碼英數。
- 報告上傳：file 必須是 PDF（非 PDF → 已知 500）。
- API：param 型別（date / int / string）。
- 反例：填字到數字欄、上傳 .txt 偽裝 .pdf。

## 2. 數值測試 `@value` / 邊界 `@edge`
- **一般值**：合法輸入（出生年 1990、手機 0912345678、代碼正確）。
- **邊界**：
  - 出生年 1800 / 2100 / 0 / 空
  - 手機 9碼 / 11碼 / 非09開頭
  - 代碼 11碼 / 13碼 / 已使用 / 不存在
  - 地址 < 6 字（已知擋）
  - 申請配額 5/5（R0，第6次擋）、0020 不計配額
  - 逾期 14 天 → 0040、30 天 → 0080、報告保留 30 天刪除
  - 訂單搜尋：起迄日相同（269）、跨月
  - 報告頁寬：A4 595 / 窄 420 / 寬 800（正規化）

## 3. 邏輯測試 `@logic`
商業規則 / 狀態機 / 分流。
- 狀態單調（245）：過 0050 不可回退；同狀態可重發、不可回朔。
- 代碼限兌換一次；已兌換(0010)不可重兌；0020 可重試。
- 路徑分流：路徑A(身分證+路徑A+OTP) vs 代碼(代碼)；本人 vs 代理人。
- OTP 三段：0000→0005→0006；帶入唯讀(272)；全欄必填擋送出(271)。
- 報告比對：OrderNo + 狀態需 0050 才可上傳。

## 4. 產檔測試 `@file`
所有輸出檔的正確性。
- **PDF**：封面 + 商品/介紹頁(單項插入、≥6頁) + QR(3組可解) + **AES-256 加密**(密碼=身分證) + 頁寬一致(A4) + 新 URL。
- **Excel**：報表匯出(兩 sheet:ID兌換+路徑B)、名單匯出(ID不隱碼)、訂單列表、月結單(隱碼)、批次範本(欄位)。
- **錯誤檔**：報告上傳錯誤紀錄 xlsx(175)。
- 反例：加密用 owner 密碼撞 `-` → qpdf 失敗（已知偶發）。

## 5. 合理性測試 `@sanity`
結果 end-to-end 對不對得上、有沒有道理。
- 報告解密後內容/封面 QR 指向 = 該人該報告。
- 名單匯出 ID = 該訂單的人。
- **會員來源(251) = 實際註冊路徑**（代碼→路徑B、路徑A→路徑A確認）。
- 月結金額/筆數對得上來源（0050/0060）。
- 統計、配額數字一致。

## 6. 其他要求綜合檢驗 `@integration`
跨切面 / 非功能 / 法遵 / 端到端。
- RBAC 權限（API 強制、角色有效）。
- 品牌用語一致（<品牌> / 實體物件）、隱碼規則。
- 法遵：同意書每次/每版簽署 + 記錄時間/人員/來源(244)。
- 整合 8 路徑 e2e；跨端(前端Web/Admin)一致。
- 資安掃描（OWASP ZAP 160 / 原碼 161）。
- iPhone 關閉 前端Web。

---

## 用法：每張票的覆蓋檢查
寫案例前，對票打勾：這票涉及哪幾類？每類要不要正例+反例？
> 例 272「ID鎖定/帶入」：@logic(分流、帶入唯讀) + @type(代碼欄英數) + @sanity(帶入的是 apply 的 ID) + @integration(兩路徑一致)。漏了 @logic 的反例(沒帶入→deadlock)就會像第一次那樣判錯。

> 例 報告 pipeline：@file(加密/頁寬/QR/插頁) + @value/@edge(頁寬 420/800) + @sanity(解密內容對人) + @logic(狀態0050才可上傳)。

# 測試維度（cross-cut 軸）—— 把上面 6 類「交叉跑」在這些軸上

矩陣思維：**測試類型 × 維度**。同一個測試類型常要在不同維度各跑一次。

## D-1 認證狀態 `@auth`（登入 / 未登入 + token/session/cookie/RBAC）
- **未登入**：打 admin API → 應 401（實測 `{"code":"UNAUTHORIZED"}`）；前端Web 未登入 → 導 LINE 登入。
- **token**：過期(跨日)→ 401；無效 token → 拒。
- **session/cookie**：前端Web Bearer 存記憶體(Pinia)不落 localStorage → 整頁重載/換人會掉；persistent profile 可續。
- **RBAC**：角色限制（下游單位帳號只能報告上傳）；API 強制權限(TICKET-XXX)。
- 反例:帶別人 token、降權角色打高權 API。

## D-2 視覺 / Responsive `@visual`（mobile / ipad / pc，最少 3 種）
- viewport：mobile 390×844、ipad 820×1180、pc 1280×800。
- 對象：前端Web 各頁(兌換驗證/填寫頁/OTP/報告) + 後台主要頁。
- 驗：版面不破、按鈕可點、文字不溢出、選項底色(深紫/淺紫)。
- 工具：Playwright projects 設不同 viewport + 截圖比對(snapshot)。
- **optional 判準**：純後台/API 不需；前台 前端Web 必跑（保戶手機為主）。

## D-3 mock / 實際串接 `@mock` / `@live`
分開測，兩邊都要：
- **@mock**：把外部依賴(外部API API / LINE / 物流商)回應 mock 掉 → 純測**我方邏輯**(狀態流、UI、錯誤處理)，不受 WAF/網路/對方資料影響、可重複。
  - 做法：Playwright route mock 前端Web 的 `/api/liff/*` 回應；或後端 server 設 mock 模式。
- **@live**：真打(後端 server 代打外部API / skl_test_v9)→ 測**真串接**(加解密、狀態單調、真 OTP)。
  - 限制：WAF auto-ban、組合會燒、UAT bypass(見 environment.md)。
- 原則：邏輯/邊界/UI 用 @mock 大量快跑；串接正確性用 @live 少量精測。

## D-4 狀態控管（狀態機，併入 @logic 但獨立追）
- 0000→0005→0006→0010 / 0030/0050/0060/0070/0080 / A0001~A0009 內部碼。
- 單調(245)、回退被擋、重進 resume、燒掉組合、配額。
- 每個轉移：正向可走、反向被擋、冪等可重發。
